# adr-craft

実装計画ではなく、設計判断を記録するADRを作り、レビューや方針変更を通じて整える。

## 基本方針

ADRには、曖昧なままだと後続の実装・データ・運用・分析の前提がずれる設計判断を残す。

PBIには、決まった方針を実装可能な作業へ分解する内容を残す。担当、順序、テーブル/カラム詳細、API、UIコンポーネント、テスト、リリース作業、具体クラス名、クエリ、index、migration詳細はPBI側に寄せる。

## 作成手順

1. まず材料を読む。Epic、概念メモ、既存ADR、レビューコメント、関連コードを確認する。
2. ADRのスコープを2つに分けて書く。
   - このADRで決めること
   - このADRで決めないこと
3. 用語集には、レビューで意味が揺れそうな語だけを載せる。
4. 決定事項は4-5個に圧縮する。自明な内容は大きな決定に統合する。
5. 各決定には、採用する方針、ドメイン上の意味、後続作業が前提にする契約を含める。
   ただし、同じ小見出しを毎回繰り返さなくてよい。読みやすい場合は、本文や箇条書きに自然に織り込む。
6. 実装方法は、その方法自体が設計判断でない限りPBIへ移す。
7. 不採用案とTODOを分ける。
8. レビュー指摘は、重要度ラベルではなく中身で分類する。

## スコープ判定

ADRに残す:
- ドメインモデルの境界
- 状態モデルとライフサイクルの意味
- 現在状態、理由、履歴、操作種別の分離
- ユーザー操作、システム操作、後続自動処理の優先関係
- 現在状態と履歴を分ける、などの論理データモデル
- 冪等性や一意性をドメイン不変条件として扱う判断
- 権限方針や信頼境界
- 後続設計に影響するrollout/計測方針
- トレードオフを踏まえて不採用にした案

PBI/実装へ移す:
- 物理テーブル名、カラム型、index、FK詳細
- SQL、Repository、UseCase、Subscriber、Batch Executor、クラス名
- migration順序やスクリプト
- UIコンポーネント構成や文言
- テスト配置やテストケース列挙
- lock/upsert/query方針。ただしそれ自体が設計判断ならADRに残す
- dashboard queryやKPI算出の実装詳細

「どう実装するか」を書いている文は外す。「後続作業が何を真として扱うべきか」を書いている文は残す。

## 推奨構成

基本形:

```markdown
# ADR: <決定領域>

# 目的
<このADRが扱うEpic範囲>

## このADRで決めること / 決めないこと
<短い箇条書き>

## レビューで特に確認してほしい点
<合意が必要な点だけ>

# 背景・制約
<判断に必要な最小限の前提>

## 用語集
<意味が揺れる語だけ>

# 決定
## 1. <決定>
...

# 検討事項
## 不採用: <案>
<なぜあり得たか、なぜ採用しないか>

# TODO
<ADR外で確認すべきこと>
```

ADRが要件定義そのものを目的にしていない限り、EpicのFR/NFR一覧を長く再掲しない。

## 書き方

- 決定を先に、理由を後に書く。
- 決定ごとの本文は、まず結論を短く書き、その後に影響範囲を説明する。
- 「ドメイン上の意味」「後続作業が前提にする契約」などの見出しは、論点が複雑で整理に効く場合だけ使う。
- 1つの概念には1つの言葉を使う。1語が2つの意味を持つなら片方を改名する。
- 「実装設計で決める」を何度も書かない。冒頭の「決めないこと」に集約する。
- テーブルは、状態モデルや論理データモデルなど、曖昧さが減る場合だけ使う。
- ドメイン語の「却下」などを、検討事項の分類語として使わない。`不採用:` を使う。
- TODOは、後でユーザーが誰に何を聞けばよいか思い出せる粒度で書く。
- 中核方針が変わったら、スコープ、用語集、決定、検討事項、TODOをまとめて見直す。

## レビュー指摘の扱い

各レビューコメントに対して:

1. 実際の懸念を抽出する。
2. `Critical` / `Suggestion` / `Nit` などのラベルは一旦無視する。
3. 中身を分類する。
   - ADRで決めるべき設計判断
   - PBI/実装詳細
   - 要件/プロダクト確認
   - 検討事項に残すべき不採用案
   - 文言修正
   - 意図的にスコープ外
4. 有用な最小変更だけ反映する。
5. 反映するとスコープが広がる場合は、TODOに移すかADR外で扱う理由を説明する。

レビュー指摘は、そのまま採用しない。既にADRで採用した方針と逆向きの指摘は、まず「不採用案として残すべきか」「TODOに逃がすべきか」「本当に決定を変えるべきか」を判定する。

ユーザー操作、システム操作、後続自動処理が同じ対象に作用する場合は、操作ごとに後続処理へ与える影響を分けて定義する。「ユーザー操作だから常に優先」「自動処理だから常に上書き可」と決めつけない。

レース条件や同時実行の指摘は、必ず防ぐ対象とは限らない。実装・運用コストと業務上の違和感を比較し、ADRでは「許容する」という決定も選択肢に含める。

物理削除、migration順序、lock方式、index、FKなどは原則PBI/実装詳細とする。ただし、ドメイン上のライフサイクルや一意性を変える場合だけADRに論理契約として残す。

返答パターン:

```text
指摘の中身は有用です。ただしADRで決めるべきなのは <方針> であり、<実装方法> ではないと整理しました。
ADRには <設計判断としての契約> を残し、<実装詳細> はPBI側で扱います。
```

## よくある失敗

- ADRがEpicの再掲になる: FR/NFR一覧を削り、判断に必要な制約だけ残す。
- ADRがPBIになる: ファイル名、SQL、index、クラス名、リリース手順を削る。
- レビュー指摘で実装詳細が戻る: 設計判断としての契約だけ残し、実装方法は落とす。
- 現在状態、理由、履歴、操作種別が混ざる: 何の値が現在の扱いを表し、何の値が「なぜそうなったか」を表すのかを分ける。
- 未確認事項を決定として書く: TODOまたはレビュー確認点へ移す。
- 不採用案が未決に見える: `不採用:` と明示し、トレードオフを書く。
