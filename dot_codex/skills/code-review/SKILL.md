# code-review

レビューだけを行う。実装、通知、PR 作成は行わない。

## レビュー体制

- `single`: 既定。本体だけで全観点を徹底的にレビューする
- `double`: 本体と subagent 1体で観点を分担し、本体が重複を除いて統合する

指定がなければ `single` とする。「徹底的に」「大規模」などの表現だけで `double` にしない。

`double` では差分に応じて subagent の役割を選ぶ。たとえば本体が正しさと設計を確認し、subagent が一貫性、テスト、安全性のうち関連する観点を受け持つ。subagent の報告をそのまま出力せず、本体が根拠を確認する。

## レビュー手順

1. ユーザーが指定した PR、差分範囲、比較元をレビュー範囲として優先する。
2. 指定がなければ PR とリポジトリの状態から対象差分を特定する。安全に特定できない場合だけ質問する。
3. 適用されるリポジトリの指示、タスク資料、仕様、参考 PR を先に読む。
4. `memo/learnings` が存在する場合は、変更箇所やタスクに関連する資料だけを読んで観点へ反映する。全件を無条件に読み込まない。現在の仕様とリポジトリの指示を優先する。
5. 差分だけで判断せず、必要な呼び出し元、近傍実装、テスト、設定、公開 API への影響を確認する。
6. 具体的な指摘を出し切るまで、正しさ、設計、命名、一貫性、テスト不足、不要コード、保守性、性能、軽微な改善を確認する。

レビュー体制や注目点を定型質問しない。不明点がレビュー結果を変える場合だけ質問する。

## 安全性レビュー

差分とコンテキストから必要性を判断し、関連する場合は安全性の観点を自動的に追加する。別引数は要求しない。

主な対象:

- 認証 / 認可 / テナント境界
- 信頼できない入力 / 解析 / デシリアライズ
- シークレット / 認証情報 / 設定
- SQL / シェル / プロセス実行
- ファイルパス / 権限 / unsafe 操作
- 外部通信 / webhook / リダイレクト
- 暗号 / トークン / セッション

`single` では本体が確認する。`double` ではリスクに応じて subagent の観点へ安全性を含める。

## 検証

PR がある場合は `gh pr view --json` で CI 状態を確認する。CI が緑、またはユーザーが CI で担保済みと示した場合は、ローカルでテストや静的解析を再実行しない。

未コミットの変更がある場合はレビュー結果の後で簡潔に知らせる。CI の SHA とローカル差分の厳密な照合は必須にしない。

CI がない、または失敗中でも、網羅的な検証を機械的に実行しない。指摘の妥当性確認に必要な場合だけ対象を絞って実行する。

## 出力形式

通常は次の枠で返す:

- `Blocking`
- `Non-blocking`

件数上限を設けない。軽微な命名、可読性、一貫性、テストの読みやすさも、今の差分で直す価値があり具体的に修正できるなら `Non-blocking` に含める。

`Blocking` と `Non-blocking` には、投稿可能で具体的なレビューコメントだけを書く。次は出力しない:

- 実装が妥当だと確認できた調査結果
- 修正を求めない subagent の報告
- 実行していないテストや検証
- 具体的な変更要求ではない前提や注意点
- 指摘として検討したが、根拠不足や誤検知として取り下げた候補

レビュー範囲の制限、未コミットの変更、外部依存など、結果の解釈を変える注意事項だけは末尾に短く添える。対応が必要なら注意事項ではなく具体的な指摘にする。指摘が1件もなければ、明示的に `LGTM` と返す。
